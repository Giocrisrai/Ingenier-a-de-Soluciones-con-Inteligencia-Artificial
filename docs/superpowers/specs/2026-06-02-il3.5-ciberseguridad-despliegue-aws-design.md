# IL3.5 — Ciberseguridad y Despliegue en AWS (Diseño)

- **Fecha:** 2026-06-02
- **Contexto:** RA3 del curso *Ingeniería de Soluciones con Inteligencia Artificial*.
- **Estado:** Aprobado para implementación.

## 1. Objetivo

Añadir a RA3 una capa hoy ausente: **ciberseguridad a nivel de aplicación/infraestructura**
y **despliegue real correcto**, complementando lo existente (IL3.1/3.2 observabilidad,
IL3.3 seguridad/ética del agente, IL3.4 escalabilidad/sostenibilidad).

El entregable es doble:

1. **Material didáctico reproducible** (módulo `RA3/IL3.5/`) que cualquiera pueda seguir.
2. **Despliegue real funcionando** de la app del curso en una cuenta **AWS Academy**,
   accesible por HTTPS para la presentación/demo.

## 2. Decisiones tomadas

| Decisión | Elección | Motivo |
|---|---|---|
| Artefacto a desplegar | **App Streamlit (frontend) + API FastAPI (backend con agente)** | Demuestra ingeniería de soluciones completa reutilizando RA1/IL1.4 y RA2. |
| Tipo de entregable | **Guía reproducible + despliegue real** | Cubre evidencia académica y demo en vivo. |
| Cuenta AWS | **Se asume Learner Lab** (escenario más restrictivo) | El usuario no estaba seguro; diseñar para lo más limitado garantiza que funcione en cualquier caso. |
| Profundidad de seguridad | **Esenciales prácticos** | Suficiente para el curso; se aplica en código/infra, no solo teoría. |
| Topología AWS | **1 EC2 + Docker Compose** | Única opción garantizada en Learner Lab (ECS/Fargate, App Runner, EKS, Beanstalk suelen estar bloqueados). |
| Ubicación del material | **Nuevo `RA3/IL3.5/` + carpeta raíz `deploy/`** | Separa material didáctico del artefacto desplegable. |

### Restricciones conocidas de AWS Academy Learner Lab
- Solo se puede usar el rol predefinido `LabRole` (no crear roles/usuarios IAM nuevos).
- Créditos limitados (~$50-100); la sesión se apaga al cerrar el lab.
- Servicios gestionados de orquestación (ECS/Fargate, App Runner, EKS, Elastic Beanstalk)
  suelen estar deshabilitados o fallar. EC2 sí funciona, con límites de tipo de instancia.

## 3. Arquitectura

Una instancia **EC2** (`t3.small`, Amazon Linux 2023) con **Docker + Docker Compose**
y tres contenedores en una red interna:

```
Internet
   │  443/80 (HTTPS)
   ▼
┌─────────── EC2 (Security Group: 443/80 público, 22 solo IP del operador) ──────────┐
│  ┌──────────┐      ┌──────────────┐      ┌────────────────────────────┐            │
│  │  caddy   │─────▶│  frontend     │─────▶│  backend (FastAPI)         │            │
│  │ (proxy + │ :8501│  Streamlit    │ :8000│  agente RA2 + RAG          │            │
│  │  HTTPS)  │      │  (RAG UI)     │      │  /chat /health /metrics    │            │
│  └──────────┘      └──────────────┘      └────────────────────────────┘            │
│         red interna docker (8501/8000 NO se exponen al host)                        │
└────────────────────────────────────────────────────────────────────────────────────┘
            │ egress HTTPS                    │ egress HTTPS
            ▼                                 ▼
      GitHub Models API                 LangSmith (observabilidad RA3.1/3.2)
```

**Flujo:** usuario → Caddy (TLS) → Streamlit → backend FastAPI por red interna → el
backend ejecuta agente/RAG, aplica guardrails de entrada/salida, llama a GitHub Models
y emite trazas a LangSmith. Solo Caddy se expone; frontend y backend nunca son
accesibles directamente desde Internet.

## 4. Componentes nuevos (`deploy/`)

| Componente | Archivo | Responsabilidad |
|---|---|---|
| Backend API | `deploy/backend/app.py` | FastAPI: `POST /chat` (envuelve un agente de RA2), `GET /health`, `GET /metrics`. Valida input con Pydantic, guardrails y rate limiting. |
| Frontend | `deploy/frontend/` | Streamlit que consume el backend (reutiliza lógica RAG de RA1/IL1.4, llamando a la API en vez de invocar el modelo directo). |
| Proxy | `deploy/Caddyfile` | TLS automático, cabeceras de seguridad (HSTS, CSP básica), reverse proxy, redirección 80→443. |
| Orquestación | `deploy/docker-compose.prod.yml` | Los 3 servicios + healthchecks + `restart: unless-stopped` + límites de CPU/mem. |
| Imágenes | `deploy/backend/Dockerfile`, `deploy/frontend/Dockerfile` | Multi-stage, usuario no-root, `uv` para deps. |
| Bootstrap | `deploy/bootstrap-ec2.sh` | Script idempotente: instala Docker, clona repo, crea `.env`, levanta el compose en una EC2 limpia. |
| Config | `deploy/.env.example` | Variables del despliegue (sin secretos reales). |

**Principio de aislamiento:** cada unidad tiene un propósito claro e interfaz definida.
El backend expone una API HTTP estable; el frontend solo conoce esa API; Caddy solo
conoce al frontend. Se pueden cambiar internos de cada uno sin romper a los demás.

## 5. Esenciales de ciberseguridad (aplicados, no solo teóricos)

### Nivel aplicación / LLM (OWASP LLM Top 10)
- **LLM01 Prompt Injection** → validación + sanitización de input en backend (refuerza
  IL3.3), system prompt blindado, lista de patrones bloqueados.
- **LLM02 Insecure Output Handling** → no renderizar HTML crudo del modelo en Streamlit; escapado.
- **LLM04 Denial of Service** → rate limiting por IP (`slowapi`) + límite de tokens/longitud de input.
- **LLM06 Sensitive Info Disclosure** → el backend nunca devuelve trazas/errores internos
  al cliente; redacción de PII en logs.

### Nivel infra / app
- **Secretos:** `.env` fuera del repo (ya en `.gitignore`), inyectado en runtime; opción
  SSM Parameter Store. Cero claves hardcodeadas.
- **Mínimo privilegio (IAM):** uso de `LabRole`; Security Group restrictivo (22 solo a la
  IP del operador, no `0.0.0.0/0`).
- **Transporte:** HTTPS forzado vía Caddy + HSTS; redirección 80→443.
- **Contenedores:** usuario no-root, imágenes slim, `read_only` donde aplique, sin puertos
  internos expuestos al host.
- **Higiene:** healthchecks, logs estructurados, checklist de "qué apagar para no gastar créditos".

## 7. Paso a paso de AWS (esqueleto a detallar en la guía)

1. Identificar la cuenta (cómo saber si es Learner Lab) y abrir el lab → "Start Lab" → consola.
2. Crear par de claves SSH + Security Group endurecido.
3. Lanzar EC2 `t3.small` Amazon Linux 2023 con `LabRole`.
4. Conectarse por SSH y ejecutar `bootstrap-ec2.sh`.
5. Cargar secretos (`.env` / SSM) y levantar `docker-compose.prod.yml`.
6. Verificar HTTPS, `/health` y demostrar el chat.
7. Probar mitigaciones (intento de injection bloqueado, rate limit 429).
8. **Apagar/terminar recursos** para no consumir créditos + nota sobre qué se pierde al
   cerrar la sesión.

## 8. Verificación

- **Local:** `docker compose -f deploy/docker-compose.prod.yml up` y probar `/health`,
  `/chat` y la UI antes de tocar AWS.
- **Seguridad:** casos de prompt injection bloqueados, rate limit devuelve 429, secretos
  ausentes en logs.
- **Notebook:** ejecuta de principio a fin sin claves reales (modo demo/mock cuando no hay token).
- **AWS:** URL HTTPS pública responde y la demo del chat funciona end-to-end.

## 9. Fuera de alcance (YAGNI)

WAF, CI de seguridad (SAST/Dependabot), multi-instancia/escalado horizontal y dominio
propio. Son del nivel "production-grade" descartado; se pueden añadir encima más adelante.
